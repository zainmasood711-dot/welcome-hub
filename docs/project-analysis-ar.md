# وثيقة تحليل المشروع — Energie Connect

**تاريخ التحليل:** 2026-06-09  
**نطاق التحليل:** الواجهة + وظائف الخادم + قاعدة البيانات + التشغيل + المخاطر

---

## 1) وصف المشروع

منصة تشغيلية عربية (RTL) لإدارة أعمال الصيانة والدعم الميداني في شركات الطاقة الشمسية، وتغطي دورة العمل من:
- إدارة الكتالوج الفني (فئات/علامات/منتجات/أكواد أعطال)
- إدارة العملاء وأنظمتهم المركبة
- إدارة التذاكر وحلولها (عن بُعد/ميداني)
- إدارة التكليفات الميدانية والتقارير
- إدارة قاعدة المعرفة وقياس فاعليتها
- لوحات مؤشرات وتقارير تشغيلية وإدارية

**الفئات المستهدفة:**
- مهندس دعم (Support Engineer)
- مهندس ميداني (Field Engineer)
- مدير (Manager)

---

## 2) الأهداف الرئيسية

1. توحيد إدارة البلاغات والمهام الميدانية في نظام واحد.
2. تسريع حل الأعطال عبر اقتراحات معرفية عملية.
3. تحسين إنتاجية فرق الميدان مع دعم عملي للوضع غير المتصل (Offline-friendly v1).
4. توفير رؤية تشغيلية وإدارية عبر تقارير ولوحات عربية واضحة.

## 8) المتطلبات الوظيفية (Functional Requirements)

1. **المصادقة والدخول:** تسجيل دخول بالبريد/الهاتف + كلمة مرور.
2. **التحكم بالصلاحيات:** تفعيل فعلي حسب الأدوار على الواجهة والخادم وقاعدة البيانات.
3. **إدارة البيانات المرجعية:** CRUD كامل للمهندسين/الكتالوج/الأخطاء.
4. **إدارة التذاكر:** إنشاء، فلترة، إسناد، حل، ربط معرفة، مرفقات.
5. **العمليات الميدانية:** إنشاء مهمة، تنفيذها، رفع الأدلة، تسليم تقرير.
6. **قاعدة المعرفة:** إنشاء/بحث/ربط/تقييم/قياس فعالية.
7. **التقارير:** مؤشرات دعم، ميدان، وإدارة.
8. **دعم انقطاع الاتصال (v1):** حفظ محلي + Queue + Sync بسيط.

---

## 9) المتطلبات غير الوظيفية (Non-Functional)

- واجهة عربية RTL متسقة
- قابلية استخدام جيدة على الموبايل خصوصًا لمسار المهندس الميداني
- أداء مقبول في القوائم والتقارير ضمن أحجام بيانات v1
- استقرار ومتانة التعامل مع فشل الشبكة
- أمن وصول قائم على RLS وليس على واجهة فقط

## 10) المشاكل المحتملة والمخاطر مع حلول مقترحة

## مخاطر عالية

1. **عدم توافق مؤشرات المعرفة مع المخطط (Schema Mismatch)**
   - **الملاحظة:** الكود يحاول تحديث `partial_count` في `knowledge_base` لكن العمود غير ظاهر في migrations الحالية.
   - **الأثر:** فشل runtime عند جلب تفاصيل المقال أو تحديث المقاييس.
   - **الحل:** إضافة migration لإضافة `partial_count` أو إزالة التحديث/الاعتماد عليه.
   - **مرجع:** `src/lib/phase2.functions.ts` (حوالي الأسطر 1422).

2. **نقطة اختناق في البحث المعرفي عند نمو البيانات**
   - **الملاحظة:** جلب حتى 120 مقال ثم ترتيب داخل التطبيق (JS ranking) بدل بحث نصي مفهرس في DB.
   - **الأثر:** تدهور الأداء مع نمو قاعدة المعرفة.
   - **الحل:** نقل ranking الأساسي للاستعلام (SQL ranking / FTS) مع فهارس مناسبة (`GIN`/`tsvector`).

## مخاطر متوسطة

3. **Offline Queue محدود عمليًا (مقصود v1 لكن يحتاج توثيق)**
   - لا يدعم رفع الصور أثناء offline (يمنع الإرسال إذا صور غير مرفوعة).
   - **الحل:** توضيح سلوك المنتج في UX + إتاحة “حفظ تقرير بدون صور” أو “إعادة رفع لاحق”.

4. **إدخال مسار مرفقات يدويًا**
   - صفحة المرفقات تسمح بإدخال `file_path` يدويًا عند عدم الرفع.
   - **الأثر:** metadata قد تشير لملف غير موجود فعليًا.
   - **الحل:** فرض الرفع الفعلي أو تحقق وجود الملف قبل حفظ metadata.

5. **اعتماد جزئي على client-side role checks للتوجيه**
   - جيد UX، لكن يجب التأكيد أن الحماية النهائية عبر RLS/ServerFns (وهو متحقق غالبًا).
   - **الحل:** الاستمرار في تدقيق كل ServerFn لصلاحية التنفيذ الحساسة (خاصة عمليات write الحرجة).

## مخاطر منخفضة

6. **تجربة واجهة متباينة لغويًا في بعض المواضع**
   - ظهور نصوص إنجليزية في الصفحة الرئيسية.
   - **الحل:** توحيد اللغة العربية في كل الشاشات النهائية.

7. **تحليل التقارير يتم على الذاكرة**
   - تجميعات `getOperationsReport` تعتمد على سحب بيانات ثم حسابها في التطبيق.
   - **الحل:** عند النمو، نقل التجميعات الثقيلة إلى SQL views/materialized views.

## 11) نتائج التحليل (Executive Findings)

1. **المشروع متقدم وظيفيًا** ويغطي دورة الدعم والميدان والمعرفة بالكامل تقريبًا لنسخة v1.
2. **الهيكل التقني سليم**: فصل واضح بين الواجهة ووظائف الخادم والبيانات.
3. **النضج الأمني جيد** بوجود RLS وسياسات دورية مفصلة واستخدام جدول أدوار مستقل.
4. **ميزة Offline عملية ومناسبة لـ v1** (بدون تعقيد مفرط) وتخدم السيناريو الحقيقي للميدان.
5. توجد **فجوات قابلة للعلاج السريع** (خصوصًا `partial_count` وأداء البحث المعرفي).

## 12) توصيات تنفيذية مرتبة بالأولوية

## أولوية قصوى (قبل الإطلاق النهائي)

2. مراجعة شاملة لمسار رفع/ربط المرفقات لمنع metadata غير صحيحة.
3. إضافة smoke test يدوي لمسارات: (Ticket -> Assignment -> Field Report -> Knowledge Feedback).

## أولوية عالية (بعد الإطلاق المبكر)
4. تحسين محرك البحث المعرفي إلى DB-level ranking.
5. تحسين تقارير العمليات عبر استعلامات SQL مجمّعة بدل الحساب الكامل داخل التطبيق.

## أولوية متوسطة

## 13) خلاصة نهائية

المشروع في وضع **قابل للإطلاق كنسخة v1 عملية** لفرق الدعم والميدان، بشرط معالجة نقطة **`partial_count`** ومراجعة موثوقية مسار المرفقات، ثم تنفيذ تحسينات الأداء البحثي والتحليلي تدريجيًا بعد الإطلاق.
