# وثيقة تحليل المشروع النهائية — Energie Connect

**الإصدار:** Final v1.0 (مُحدَّث)  
**تاريخ التحديث:** 2026-06-09  
**نطاق التحليل:** البنية الأساسية + التقنيات + الوظائف + التشغيل + المخاطر + الجاهزية للإطلاق

---

## 1) وصف المشروع

**Energie Connect** منصة تشغيلية عربية (RTL) لإدارة دورة الدعم الفني والميدان في شركات الطاقة الشمسية، من استقبال البلاغ وحتى إغلاق التذكرة وتسجيل المعرفة القابلة لإعادة الاستخدام.

### ما الذي يغطيه النظام؟
- إدارة الكتالوج الفني (الفئات، العلامات، المنتجات، أكواد الأعطال)
- إدارة العملاء والأنظمة المركبة ومكوناتها
- إدارة التذاكر (إنشاء، متابعة، حلول، مرفقات)
- إدارة التكليفات الميدانية وتقارير التنفيذ
- إدارة قاعدة المعرفة وقياس فاعلية الحلول
- لوحات معلومات وتقارير تشغيلية وإدارية

### المستخدمون المستهدفون
- مهندس دعم
- مهندس ميداني
- مدير/مشرف عمليات

---

## 2) الأهداف الرئيسية للمشروع

1. توحيد عمليات الدعم والميدان داخل نظام واحد واضح.
2. تقليل زمن الحل عبر قاعدة معرفة مرتبطة بالتذاكر والتقارير.
3. توفير تجربة ميدانية عملية على الموبايل مع دعم Offline-friendly (نسخة v1).
4. تقديم رؤية تشغيلية وإدارية عبر لوحات وتقارير عربية قابلة للتنفيذ.

---

## 3) التقنيات المستخدمة والبنية الأساسية

### واجهة التطبيق
- **TanStack Start (React 19 + TypeScript)**
- **TanStack Router** (File-based routing)
- **TanStack Query** لإدارة الجلب والكاش
- **Tailwind CSS v4** + مكتبة مكونات UI
- **Recharts** للرسوم البيانية
- **Sonner** للتنبيهات

### الخادم والبيانات
- وظائف خادم عبر `createServerFn` (نمط RPC typed)
- Middleware للمصادقة والتحقق من الصلاحيات
- PostgreSQL مع سياسات أمان RLS
- تخزين ملفات الميدان داخل bucket للمرفقات

### المرجع التقني الأساسي
- `package.json`
- `src/router.tsx`, `src/start.ts`
- `src/lib/phase1.functions.ts`, `src/lib/phase2.functions.ts`
- `src/integrations/supabase/auth-middleware.ts`
- `supabase/migrations/*.sql`

## 4) المتطلبات التشغيلية

### متطلبات البيئة
- Bun/Node حديث
- متغيرات البيئة الأساسية:
  - `SUPABASE_URL`
  - `SUPABASE_PUBLISHABLE_KEY`
  - `VITE_SUPABASE_URL`
  - `VITE_SUPABASE_PUBLISHABLE_KEY`
  - `SUPABASE_SERVICE_ROLE_KEY` (للعمليات الإدارية والـ seed)

### تشغيل المشروع
- `bun install`
- `bun run dev`

### تشغيل قاعدة البيانات
- تطبيق جميع ملفات migration بالترتيب
- التأكد من تفعيل RLS والسياسات والفهارس
- استخدام seed التجريبي للمراجعة السريعة (عند الحاجة)

## 5) هيكل المشروع (تفصيلي)

### مسارات الواجهة
- عامة: `/`, `/auth`
- محمية: `/_authenticated/*`
  - Dashboard
  - Engineers / Catalog / Error Codes
  - Customers / Customer Systems
  - Tickets
  - Assignments + تفاصيل المهمة
  - Field Task (نسخة موبايل)
  - Attachments
  - Knowledge Base + تفاصيل المقال
  - Notifications
  - Reports + صفحات تقارير متخصصة

### الطبقات المعمارية
1. **UI Layer**: شاشات وتجارب المستخدم داخل `src/routes/_authenticated/*`
2. **Server Functions Layer**: منطق الأعمال داخل `src/lib/phase1.functions.ts` و`src/lib/phase2.functions.ts`
3. **Data/Security Layer**: مخطط قاعدة البيانات + السياسات + الفهارس عبر migrations

---

## 6) الوظائف الرئيسية المتاحة حاليًا

### أ) إدارة الوصول والصلاحيات
- إدارة سياق الوصول حسب الدور
- توجيه الواجهة حسب الدور
- فرض الوصول على مستوى الخادم/البيانات عبر السياسات

### ب) العمليات الأساسية (Phase 1)
- إدارة المهندسين
- إدارة الكتالوج الفني
- إدارة أكواد الأعطال

### ج) العمليات التشغيلية (Phase 2)
- العملاء، الأنظمة، المكونات
- التذاكر وحلولها ومرفقاتها
- التكليفات الميدانية (إنشاء من تذكرة/نظام)
- تنفيذ المهام الميدانية ورفع التقارير
- قاعدة المعرفة (إنشاء/بحث/تفاصيل/تغذية راجعة/مؤشرات)
- لوحات وتقارير تشغيلية وإدارية

### د) تجربة الميدان Offline-friendly (v1)
- حفظ مسودة التقرير محليًا
- كاش للمهام المسندة وتفاصيل المهمة
- مؤشر اتصال مباشر (Online/Offline)
- طابور إرسال مبسط للمحاولات الفاشلة مع مزامنة عند عودة الشبكة
- تحسين رفع صور الموبايل:
  - ضغط قبل الرفع
  - إعادة محاولة تلقائية عند الفشل
  - شريط تقدم واضح قبل إرسال التقرير

## 8) المتطلبات الوظيفية (Functional)

1. مصادقة المستخدمين والدخول الآمن.
2. تطبيق الصلاحيات حسب الدور على كل من الواجهة والخادم والبيانات.
3. إدارة كاملة للبيانات المرجعية (CRUD).
4. دورة تذاكر مكتملة (إنشاء، متابعة، إسناد، حل، إغلاق).
5. دورة تكليف ميداني مكتملة (تنفيذ، أدلة، تقرير).
6. دورة معرفة مكتملة (إنشاء، بحث، ربط، تقييم، قياس فاعلية).
7. لوحات وتقارير تشغيلية وإدارية.
8. دعم عملي للانقطاع الشبكي في مسار الميدان (v1).

---

## 9) المتطلبات غير الوظيفية (Non-Functional)

- واجهة عربية RTL واضحة ومتسقة
- قابلية استخدام عالية على الموبايل لمسار المهندس الميداني
- أداء جيد ضمن حجم بيانات v1
- رسائل حالة/أخطاء واضحة أثناء التحميل والرفع والمزامنة
- أمان يعتمد على سياسات البيانات لا على الواجهة فقط

---

## 10) المخاطر المحتملة والحلول المقترحة

### مخاطر عالية
1. **احتمال عدم توافق حقول مقاييس المعرفة (`partial_count`) مع المخطط الفعلي**
   - **الأثر:** أخطاء وقت التنفيذ في تفاصيل المقال/تحديث المقاييس.
   - **الحل:** مراجعة نهائية للمخطط + Migration تصحيحية (إضافة العمود أو تعديل الكود).

2. **قابلية تدهور أداء بحث قاعدة المعرفة مع نمو البيانات**
   - **الأثر:** بطء نتائج البحث عند التوسع.
   - **الحل:** نقل الترتيب الأساسي إلى DB-level Full-Text Search مع فهارس `GIN/tsvector`.

### مخاطر متوسطة
3. **الـ Offline في v1 لا يغطي رفع الصور بدون اتصال بالكامل**
   - **الحل:** توضيح سلوك المنتج للمستخدم + خيار إكمال التقرير ثم رفع الصور لاحقًا (إذا تقرر).

4. **إمكانية تسجيل metadata مرفقات لمسار غير موجود فعليًا**
   - **الحل:** فرض التحقق من الملف المرفوع قبل حفظ المرجع.

5. **التقارير الثقيلة تُحسب جزئيًا على الذاكرة**
   - **الحل:** نقل التجميعات الثقيلة مستقبلًا إلى SQL views/materialized views.

### مخاطر منخفضة
6. **وجود نصوص غير عربية في بعض الشاشات**
   - **الحل:** تدقيق لغوي نهائي وتوحيد المصطلحات العربية.

---

## 11) نتائج التحليل النهائية

1. المشروع يغطي معظم المتطلبات التشغيلية الأساسية لنسخة v1.
2. البنية المعمارية واضحة وقابلة للتطوير.
3. مستوى الأمان جيد بوجود سياسات وصول واضحة على مستوى البيانات.
4. تجربة الميدان أصبحت أكثر عملية واستقرارًا (خصوصًا في الموبايل والرفع).
5. توجد نقاط تحسين محددة وواضحة يمكن تنفيذها بعد الإطلاق المبكر دون تعطيل العمل.

---

## 12) خطة أولوية التنفيذ (بعد التحديث)

### قبل الإطلاق النهائي مباشرة
1. حسم وضع `partial_count` بشكل نهائي (مخطط + كود).
2. التحقق من صحة دورة المرفقات من الرفع حتى التخزين المرجعي.
3. تنفيذ اختبار سيناريو شامل:
   - Ticket → Assignment → Field Report → Knowledge Feedback

### بعد الإطلاق المبكر
4. تطوير البحث المعرفي إلى Full-Text Search على مستوى قاعدة البيانات.
5. ترحيل التجميعات التحليلية الثقيلة من الواجهة إلى استعلامات مجمعة.

### تحسينات تشغيلية لاحقة
6. توحيد اللغة العربية في جميع الرسائل والعناوين.
7. إضافة قياس خفيف لمعدلات فشل الرفع/المزامنة لمتابعة الجودة ميدانيًا.

---

## 13) الخلاصة التنفيذية

المشروع **جاهز عمليًا كنسخة v1** للاستخدام التشغيلي في الدعم والميدان، مع توصية بإغلاق نقاط المراجعة الحرجة (مطابقة مقاييس المعرفة + موثوقية المرفقات) لضمان إطلاق مستقر وعالي الثقة.
